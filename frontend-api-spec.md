# AI Customer Service Platform — Frontend API Spec

Source-of-truth doc for wiring a React dashboard against the existing Java/Spring backend.
Generated by reading PRD.txt, ARCH.txt, every controller, every request/response DTO, and every `application.properties` / `service.properties`.

---

## 1. Product Context

### What this product does (5 bullets)
- **24/7 AI phone agent** — small businesses get an extra Twilio phone number; when a customer calls, an AI answers using only that business's knowledge.
- **Strict multi-tenant isolation** — Business A's data, knowledge, and call logs are never accessible to Business B; enforced at every DB query (`WHERE business_id = ?`) and via JWT-claim guards.
- **Structured knowledge ingestion** — businesses configure a structured profile (hours, services, payment, policies), FAQs, free-form context, and escalation rules; the platform renders these into the LLM system prompt at call time. (File-upload PDF/OCR is Phase 2.)
- **Post-call intelligence** — every call produces a transcript, AI-generated summary, a 1/2 keypress satisfaction score, and a computed "interest rating" used to prioritise follow-ups.
- **Configurable lead scoring** — each business can tune the signals (long call, callback requested, positive feedback, etc.) that feed the interest rating.

### Dashboard user
The **small business owner / operator** (e.g. dentist, salon, repair shop). NOT the end customer who calls in. PRD §2 names them `Business`. They are the single human per business in MVP; multi-user support is out of scope (a `business_users` table is foreshadowed but not built).

### Core jobs-to-be-done for the dashboard
1. **Sign up / sign in** (POST /business/register, POST /auth/signin).
2. **Set up the business profile** so the AI sounds correct — name, category, location, operating hours.
3. **Onboard knowledge** — fill out structured profile, add FAQs, paste free-form notes, define escalation rules. See a *completeness score* with missing-field nudges.
4. **Manage Twilio numbers** assigned to the business (list, add, remove).
5. **Configure rating signals** to tune which calls bubble to the top.
6. **Review recent calls** — name, phone, summary, duration, time, interest rating, feedback. (Last 24h focus per PRD §3.4.1; backend currently returns a flat "recent" list — see §5.)
7. **Work the callback queue** — sorted earliest-first list of calls where AI couldn't answer or customer asked for a human.
8. **Account / settings** — view profile, change basic fields.

## 2. Tenant-facing API Contracts

### Conventions used throughout

- **Base URL pattern (local dev):** `http://localhost:{port}/{service-name}/api/v1/...`
  Every service mounts itself under a context path equal to its name (see §4).
- **Auth header:** `Authorization: Bearer <accessToken>`
- **JSON content type:** `application/json` for both request and response.
- **IDs are ULIDs** (26-char base32 strings) returned as `String` — *not* UUIDs.
- **Timestamps are ISO-8601 `Instant`** strings (e.g. `"2026-05-24T09:31:02.123Z"`).
- **Tenant guard:** every `{id}` / `{businessId}` path variable must equal the `businessId` claim on the JWT, or you get HTTP 403 (`ApiError {status, message:"Access denied"}`). Platform admins (`ROLE_PLATFORM_ADMIN`) bypass this; MVP signins never have that role.
- **Scopes (granted as Spring authorities prefixed `SCOPE_`):**
  - `business.read`, `business.write` — granted by every MVP signin
  - `knowledge.read`, `knowledge.write` — **NOT in MVP signin claims (gap)** — see §5 gotchas
  - `calls.read` — **NOT in MVP signin claims (gap)** — see §5 gotchas

### 2.1 auth-service (port 8081, context `/auth-service`)

#### POST `/auth-service/api/v1/auth/signin`
- **Auth:** none (public).
- **Purpose:** Business login by email + password.
- **Request body** (`SigninRequest`):
  ```
  {
    "email":    string (required, valid email),
    "password": string (required, non-blank)
  }
  ```
- **Response 200** (`AuthenticationResponse`):
  ```
  {
    "token":        string,   // access token JWT
    "refreshToken": string,   // refresh token JWT
    "tokenType":    string,   // always "Bearer"
    "expiresIn":    number,   // seconds until access token expiry (default 900 = 15m)
    "message":      string    // "Login successful"
  }
  ```
- **Errors:** 400 validation; 401 `AuthFailedException` ("Invalid email or password" / "Business account is inactive").

#### POST `/auth-service/api/v1/auth/refresh`
- **Auth:** none — pass the refresh token in the body, not the header.
- **Purpose:** Exchange a valid refresh token for a new access token.
- **Request body** (`RefreshTokenRequest`):
  ```
  { "refreshToken": string (required) }
  ```
- **Response 200** (`RefreshTokenResponse`):
  ```
  {
    "accessToken": string,
    "tokenType":   string,   // "Bearer"
    "expiresIn":   number,   // seconds (default 900)
    "message":     string    // "Access token refreshed successfully"
  }
  ```
- **Errors:** 401 invalid/expired refresh token.

> There is no `/auth/logout` endpoint and no `/auth/validate` endpoint, despite ARCH §2.2 listing them. Logout is purely client-side (drop the tokens).

### 2.2 user-business-service (port 8082, context `/user-business-service`)

#### POST `/user-business-service/api/v1/business/register`
- **Auth:** none (public).
- **Purpose:** Self-register a new business; also creates default rating-config rows.
- **Request body** (`RegisterBusinessRequest`):
  ```
  {
    "name":            string (required, max 200),
    "email":           string (required, valid email, unique),
    "password":        string (required, 8-100 chars),
    "category":        string (optional),
    "description":     string (optional),
    "location":        string (optional),
    "operatingHours":  string (optional)   // free-text in this DTO (structured hours live in knowledge-service)
  }
  ```
- **Response 201** (`BusinessResponse`):
  ```
  {
    "id":              string (ULID),
    "name":            string,
    "email":           string,
    "category":        string|null,
    "description":     string|null,
    "location":        string|null,
    "operatingHours":  string|null,
    "isActive":        boolean,
    "createdAt":       string (ISO-8601),
    "updatedAt":       string (ISO-8601)|null
  }
  ```
- **Errors:** 400 validation; 409 `ConflictException` (duplicate email).

#### GET `/user-business-service/api/v1/business/{id}/profile`
- **Auth:** JWT, scope `business.read`, `{id}` must match JWT `businessId`.
- **Response 200:** `BusinessResponse` (same shape as above).
- **Errors:** 401, 403, 404 `BusinessNotFoundException`.

#### PUT `/user-business-service/api/v1/business/{id}/profile`
- **Auth:** JWT, scope `business.write`, tenant match.
- **Request body** (`UpdateBusinessProfileRequest`) — all fields optional:
  ```
  {
    "name":           string|null (max 200),
    "category":       string|null,
    "description":    string|null,
    "location":       string|null,
    "operatingHours": string|null
  }
  ```
- **Response 200:** `BusinessResponse`.

#### GET `/user-business-service/api/v1/business/{id}/phone-numbers`
- **Auth:** JWT, scope `business.read`, tenant match.
- **Response 200:** `List<PhoneNumberResponse>`:
  ```
  [{
    "id":            string (ULID),
    "businessId":    string,
    "twilioNumber":  string (E.164, e.g. "+14155551234"),
    "label":         string|null,
    "isActive":      boolean,
    "createdAt":     string (ISO-8601)

  ```

#### POST `/user-business-service/api/v1/business/{id}/phone-numbers`
- **Auth:** JWT, scope `business.write`, tenant match.
- **Request body** (`AddPhoneNumberRequest`):
  ```
  {
    "twilioNumber": string (required, E.164 regex `^\+[1-9]\d{6,14}$`),
    "label":        string|null (max 100)
  }
  ```
- **Response 201:** `PhoneNumberResponse`.

#### DELETE `/user-business-service/api/v1/business/{id}/phone-numbers/{numberId}`
- **Auth:** JWT, scope `business.write`, tenant match.
- **Response 204** no content.

#### GET `/user-business-service/api/v1/business/{id}/rating-config`
- **Auth:** JWT, scope `business.read`, tenant match.
- **Response 200** (`RatingConfigResponse`):
  ```
  {
    "businessId": string,
    "entries": [{
      "id":         string (ULID),
      "signalKey":  string,   // LONG_CALL | POSITIVE_FEEDBACK | CALLBACK_REQUESTED | NEGATIVE_FEEDBACK | SHORT_CALL | AI_COULD_NOT_ANSWER
      "scoreValue": number (integer),
      "updatedAt":  string (ISO-8601)
    }, ...]
  }
  ```

#### PUT `/user-business-service/api/v1/business/{id}/rating-config`
- **Auth:** JWT, scope `business.write`, tenant match.
- **Request body** (`UpdateRatingConfigRequest`):
  ```
  {
    "entries": [{
      "signalKey":  string (required, non-blank),
      "scoreValue": number (required, integer; may be negative)
    }, ...]   // non-empty
  }
  ```
- **Response 200:** `RatingConfigResponse`.

> Default rating-config rows inserted at registration: `LONG_CALL=2`, `POSITIVE_FEEDBACK=2`, `CALLBACK_REQUESTED=3`, `NEGATIVE_FEEDBACK=-1`, `SHORT_CALL=-2`, `AI_COULD_NOT_ANSWER=1`.

### 2.4 call-orchestration-service (port 8086, context `/call-orchestration-service`)

`CallController` exposes exactly **one** tenant-facing endpoint today. The rest of the call data lives behind `/api/internal/**` (service-to-service only — dashboard-service would consume those if it existed). For an MVP frontend, this is what you have:

#### GET `/call-orchestration-service/api/v1/calls/{businessId}/recent`
- **Auth:** JWT, scope `calls.read`, tenant match. *(Scope not in MVP signin claims — see gotchas.)*
- **Purpose:** Recent call logs for the business. No pagination, no time filter on the controller signature (service-layer implementation decides whether it caps at last 24h or last N).
- **Response 200:** `List<CallLogResponse>`:
  ```
  [{
    "id":                 string (ULID),
    "businessId":         string,
    "customerPhone":      string|null,
    "customerName":       string|null,
    "provider":           string,             // "twilio", ...
    "providerCallId":     string,             // Twilio CallSid, unique
    "queryType":          string|null,
    "callSummary":        string|null,        // AI-generated, filled async post-call
    "transcript":         string|null,        // JSON-serialised conversation history
    "callDurationSecs":   number|null,
    "feedbackScore":      number|null,        // 1 = yes, 2 = no, null = no answer
    "interestRating":     number|null,
    "callbackRequested":  boolean,
    "callStartedAt":      string (ISO-8601),
    "callEndedAt":        string (ISO-8601)|null
  }, ...]
  ```

> No dedicated `/callbacks` endpoint is exposed publicly today (only `GET /api/internal/calls/{businessId}/callbacks` exists, requires a service token). The frontend will need to filter `callbackRequested == true` client-side from the `/recent` payload OR a new tenant endpoint must be added. **Recommend:** add `GET /api/v1/calls/{businessId}/callbacks` mirroring the internal one before shipping the dashboard.

## 3. Login Flow

1. **Login**: `POST /auth-service/api/v1/auth/signin` with `{email, password}`.
2. **Response** (`AuthenticationResponse`):
   - `token` — access token (JWT, HS512).
   - `refreshToken` — refresh token (JWT, HS512).
   - `tokenType` — `"Bearer"`.
   - `expiresIn` — seconds until access token expiry.
3. **Token lifetimes** (from `auth-service/secrets/secrets.properties`):
   - Access token: **15 minutes** (`secrets.jwt.access-token-expiration=15m`)
   - Refresh token: **7 days** (`secrets.jwt.refresh-token-expiration=7d`)
   - Service token: 15 minutes (irrelevant for frontend)
4. **Refresh**: `POST /auth-service/api/v1/auth/refresh` with `{refreshToken}`. Returns a fresh access token (15-min TTL) and the same `refreshToken` you held is still valid until 7 days from issue. (There's no rotation today — same refresh token can be reused until it expires.)
5. **businessId location in JWT**: payload claim `businessId` (ULID). The full claim shape:
   ```
   {
     "sub":        "<business email>",
     "uid":        "<businessId ULID>",   // same as businessId in MVP
     "businessId": "<businessId ULID>",
     "roles":      ["ROLE_BUSINESS_ADMIN"],
     "scopes":     ["business.read", "business.write"],
     "iat":        <epoch seconds>,
     "exp":        <epoch seconds>
   }
   ```
   Refresh tokens carry the *same* claims as access tokens (only difference is the longer expiry).

6. **Logout**: client-side only — drop the tokens. No server endpoint exists.

## 5. Gotchas

### CORS
- **Not configured anywhere.** `user-business-service/SecurityConfig` calls `.cors(AbstractHttpConfigurer::disable)` and so do the others. `auth-service/SecurityConfig` is `permitAll()` with CSRF off but no CORS bean. The browser will be blocked by preflight on every cross-origin request.
- **Recommend:** either (a) put Caddy in front and serve the React app from the same origin, or (b) add a `CorsConfigurationSource` bean to each service for `http://localhost:5173` (or wherever Vite runs). The `api-gateway` from ARCH §2.1 is not built yet.

### Authorization header
- Format: `Authorization: Bearer <accessToken>`.
- `tokenType` in the login response is the literal string `"Bearer"` — concat with the access token. The JWT filter reads the `Authorization` header, strips the `Bearer ` prefix, and verifies HS512 with the shared secret.

### Scope mismatch (real bug for the frontend)
auth-service hands out *only* `business.read` and `business.write` scopes at signin (see `AuthenticationService.MVP_SCOPES`). But:
- knowledge-service controllers demand `SCOPE_knowledge.read` / `SCOPE_knowledge.write`.
- call-orchestration-service's `CallController.recent` demands `SCOPE_calls.read`.

A real business JWT will hit **403 Forbidden** on every knowledge and calls endpoint until the MVP scope list is widened in `AuthenticationService` to include `knowledge.read`, `knowledge.write`, `calls.read`. Flag this to the backend owner before you waste an afternoon debugging it. (The `BusinessAccessGuard` tenant check works fine — the failure is the scope check.)

### Tenant guard
Every protected endpoint runs `@businessAccessGuard.canAccess(#id)` — the path `{id}` must equal the JWT's `businessId` claim. So the React app should always pull `businessId` out of the JWT (decode it client-side; no need to verify) and use it when building URLs.

### Pagination
- **None.** No `Page`, no `Pageable`, no `limit/offset` query params on any tenant endpoint. `GET .../recent` returns the full list. The frontend will need to paginate client-side until the backend adds it.

### Call summaries: where do they come from?
- `conversation-summary-service` generates them, then `PUT`s them onto the `call_logs` row in `call-orchestration-service`. The frontend never talks to summary-service. Read the `callSummary` field off `CallLogResponse` from `GET /call-orchestration-service/api/v1/calls/{businessId}/recent`.
- The summary is filled **asynchronously** post-call (a few seconds after the call ends). If you fetch `/recent` right after a call ends you may see `callSummary: null` for a moment. Poll or re-fetch.

### No callback-list tenant endpoint
There's no public `GET /api/v1/calls/{businessId}/callbacks`. Either:
- ask backend to add one, or
- filter `callLogs.filter(c => c.callbackRequested === true)` client-side.

### Time semantics
All timestamps are UTC ISO-8601. Display in the user's local timezone in the UI.

### Phone numbers
E.164 only, enforced via regex `^\+[1-9]\d{6,14}$`. The UI should ask for country code + local number and assemble before POSTing.

### Health check
Every service has a public `GET /{context-path}/api/v1/health` (no auth) returning a simple UP payload — useful for the dashboard's "service status" widget if you build one.

### Default rating-config rows
Inserted on `POST /business/register`. The signal keys the UI should be ready to render in the rating-config editor: `LONG_CALL`, `POSITIVE_FEEDBACK`, `CALLBACK_REQUESTED`, `NEGATIVE_FEEDBACK`, `SHORT_CALL`, `AI_COULD_NOT_ANSWER`. There is no metadata endpoint listing them — hardcode the labels and descriptions in the UI.

### JSON naming
- Most response DTOs use **camelCase** (Spring Jackson default).
- Two domain objects nested under `ProfileResponse` use **snake_case**: `Service` (`price_min`, `price_max`, `price_currency`, `duration_minutes`) and `BusinessHours.DayHours` keys (`open`, `close`, `closed` are already single-word). These are annotated `@JsonNaming(SnakeCaseStrategy.class)` and won't auto-deserialise into a camelCase TS type without a custom mapper.

### File upload (knowledge)
Originally specified in ARCH §2.4 (PDF/TXT/image upload to S3). **Not in MVP**. Phase 2 only. Don't build the UI for it yet.

### Error response shape
Every service returns errors as `ApiError`:
```
{ "status": number (HTTP code), "message": string }
```
401 → `"Authentication required"`, 403 → `"Access denied"` (unless an exception message overrides).
