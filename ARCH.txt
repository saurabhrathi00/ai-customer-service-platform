1. Architecture Overview

This document defines the complete microservices architecture for the AI Customer Service Platform. The system is decomposed into 11 independent services, each owning its own data, responsibilities, and deployment lifecycle. Services communicate via REST APIs synchronously and via an event/message queue asynchronously where decoupling is critical.

1.1 Architecture Principles
	•	Single Responsibility — each service does one thing and does it well.
	•	Database per Service — no shared databases between services.
	•	API Gateway as single entry point — all external traffic passes through the gateway.
	•	Strict Data Isolation — Business A's data is never accessible to Business B at any layer.
	•	Stateless Services — services do not hold session state; JWT tokens carry identity.
	•	Async where possible — long-running tasks (summary generation, notifications) are decoupled via message queue.

How Twilio Calls This Service

2.7 call-orchestration-service
Property
Detail
Port
8086 (HTTP) + WebSocket
Technology
Spring Boot + WebSocket (javax.websocket)
Responsibility
Live call management, transcript building, metadata storage, interest score calculation
Owns Database?
Yes — call_logs table (single owner)
Communicates With
ai-conversation-service, knowledge-service, conversation-summary-service, notification-service

This is the most critical service. Here is what it does:
	•	Accepts WebSocket connection from Twilio Media Stream when a call starts.
	•	Receives audio chunks from customer in real time.
	•	Sends audio to Speech-to-Text (Twilio built-in or Deepgram) to get transcript text.
	•	Sends transcript text to ai-conversation-service with business knowledge context.
	•	Receives AI text response, converts to speech (TTS), sends audio back to Twilio.
	•	Maintains full conversation transcript in memory during the call.
	•	On call end (Twilio status callback):
	•	Saves call metadata to call_logs (start time, end time, duration, customer phone, businessId).
	•	Fires transcript to conversation-summary-service (async).
	•	If callback was requested, notifies notification-service (async).
	•	Calculates interest score using business rating config.

5.1 Data Isolation Enforcement
	•	Every query to knowledge_base includes WHERE business_id = ? — never fetches all businesses.
	•	Every query to call_logs includes WHERE business_id = ? — never fetches all businesses.
	•	The Twilio number → businessId mapping is the gatekeeper — one wrong mapping could break isolation.
	•	Unit tests must verify that API calls for Business A cannot return Business B data.

6.2 Twilio Request Security
	•	All Twilio webhooks include an X-Twilio-Signature header.
	•	incoming-call-service and feedback-service must validate this signature on every request.
	•	Use Twilio Java SDK: RequestValidator.validate(url, params, signature).
	•	Requests with invalid signatures must be rejected with HTTP 403.

6.3 Number Provisioning Flow
	•	When a business onboards, user-business-service calls Twilio API to purchase a new number.
	•	The number is saved to business_phone_numbers table linked to that businessId.
	•	Twilio number's webhook URL is configured to point to: https://yourdomain.com/twilio/incoming/call.

9. Internal Service Structure — MVC Pattern

Each microservice follows the MVC (Model-View-Controller) pattern internally. The microservices architecture governs how services talk to each other at the macro level; MVC governs how code is organised inside each service at the micro level.

9.1 MVC Layer Responsibilities
Layer
Spring Boot Component
Responsibility
Controller (C)
@RestController
Handles incoming HTTP requests. Validates input. Returns HTTP responses. Contains no business logic.
Service (S)
@Service
All business logic lives here. Calls repositories and other services. Orchestrates operations.
Repository (R)
@Repository / JpaRepository
Direct database interaction. JPA queries. No business logic.
Model (M)
@Entity + DTOs
JPA Entities map to DB tables. DTOs are request/response objects for APIs.
Config
@Configuration
Spring beans, security config, Twilio client setup, S3 client setup.

9.2 Standard Folder Structure (Every Service)
src/main/java/com/aiassistant/{service-name}/ ├── controller/ │   └── BusinessController.java       ← @RestController, HTTP endpoints ├── service/ │   ├── BusinessService.java          ← @Service, business logic │   └── impl/ │       └── BusinessServiceImpl.java  ← implementation ├── repository/ │   └── BusinessRepository.java       ← @Repository, JPA queries ├── model/ │   ├── entity/ │   │   └── Business.java             ← @Entity, maps to DB table │   └── dto/ │       ├── BusinessRequestDTO.java   ← incoming request body │       └── BusinessResponseDTO.java  ← outgoing response body ├── config/ │   ├── SecurityConfig.java           ← Spring Security config │   └── AppConfig.java                ← Beans, clients ├── exception/ │   ├── BusinessNotFoundException.java │   └── GlobalExceptionHandler.java   ← @ControllerAdvice └── {ServiceName}Application.java     ← @SpringBootApplication main

9.3 Example: user-business-service MVC Flow
POST /business/register  1. BusinessController.register(@RequestBody BusinessRequestDTO dto)    → validates input, calls BusinessService  2. BusinessService.registerBusiness(BusinessRequestDTO dto)    → checks duplicate email    → hashes password (BCrypt)    → calls Twilio to provision a number    → saves entity via BusinessRepository    → returns BusinessResponseDTO  3. BusinessRepository.save(Business entity)    → JPA persists to Supabase PostgreSQL  4. BusinessController returns HTTP 201 + BusinessResponseDTO

9.4 DTO vs Entity — Rule
	•	Entities are NEVER returned directly from controllers — always convert to DTO.
	•	Entities contain DB-specific annotations (@Entity, @Column, @Id).
	•	DTOs contain only what the API consumer needs — no passwords, no internal IDs leaked.
	•	Use a mapper class or ModelMapper library to convert between Entity and DTO.

11.3 How Each Service Deploys on Railway
	•	Each Spring Boot service has a Dockerfile in its root.
	•	Connect the GitHub repo to Railway — it auto-detects the Dockerfile.
	•	On every git push to main branch → Railway automatically rebuilds and redeploys.
	•	Environment variables (API keys, DB passwords) are set in Railway's dashboard — never in code.
	•	Each service gets a private Railway internal URL (e.g. api-gateway.railway.internal:8080).
	•	Only api-gateway gets a public domain exposed to the internet.
